# Using the Cluster Proxy

Single consumer-facing reference for the cluster's OpenAI-compatible LiteLLM proxy. If you are integrating a tool against Foundation's local inference, this is the page to read first.

## Base URL

```
http://10.2.35.10:4000/v1
```

- **API key:** `none` (the proxy does not require a real key; pass `"none"` or any non-empty placeholder where your client demands one).
- The proxy runs on the cluster **master** (aivm, `10.2.35.10`) and routes by `model` name to the right backend node.

## Models

| Served name | Use for | Backend | Max context |
|---|---|---|---|
| `gemma-4-26b-a4b-nvfp4` | Chat / completions | GPU Server 1 (`10.2.35.30:8020`) | 100K |
| `nomic-embed-text-v1-5` | Embeddings | Death Star (`10.2.35.20:8022`) | 2048 |

- `gemma-4-26b-a4b-nvfp4` is instruction-tuned (NVFP4 quant) and works natively via `/v1/chat/completions`.
- `nomic-embed-text-v1-5` returns **768-dim** vectors. Its max input context is **2048 tokens** (not 8192, as is commonly assumed) — chunk longer inputs before sending.

## Chat / completions

Standard OpenAI-compatible request:

```bash
curl http://10.2.35.10:4000/v1/chat/completions \
  -H "Authorization: Bearer none" \
  -H "Content-Type: application/json" \
  -d '{
    "model": "gemma-4-26b-a4b-nvfp4",
    "messages": [{"role": "user", "content": "Hello"}]
  }'
```

## Embeddings

A **bare** request works — `encoding_format` is pinned to `float` proxy-side, so consumers no longer need to send it themselves:

```bash
curl http://10.2.35.10:4000/v1/embeddings \
  -H "Authorization: Bearer none" \
  -H "Content-Type: application/json" \
  -d '{
    "model": "nomic-embed-text-v1-5",
    "input": "your text here"
  }'
```

> **Background:** LiteLLM injects `encoding_format=None` on `/v1/embeddings`, which vLLM rejects with HTTP 400. As of 2026-06-30 the cluster config generator pins `encoding_format: float` on every embedding model, so bare calls succeed. (Previously, every consumer had to pass `encoding_format: "float"` explicitly.) Keep the input under the 2048-token embedding context limit.

## How the config is managed

The proxy's `litellm/cluster_config.yaml` is **auto-generated by the control agent** (`agent/agent.py`) on the master and is marked "do not edit by hand." Any change you make directly will be overwritten on the next sync.

To change proxy behavior, edit the **generator** (`agent/agent.py`), then redeploy on the master:

```bash
git pull --ff-only
bash agent/stop_agent.sh && bash agent/start_agent.sh   # agent on :5000
curl -X POST http://127.0.0.1:5000/proxy/sync           # regenerate config + bounce :4000
```

The `/proxy/sync` step bounces the `:4000` proxy (a few-second chat blip).
